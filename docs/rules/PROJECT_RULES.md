## 🤖 Правила для ботов

### Правило 1: BAT-файлы для запуска ботов

**Для каждого бота ОБЯЗАТЕЛЬНО создавать BAT-файл для запуска**

#### Требования:

1. **Расположение:** `scripts/` директория
2. **Именование:** `run_<bot_name>.bat`
3. **Кодировка:** UTF-8 (для корректного отображения кириллицы)
4. **Содержимое:** Минимум - активация venv и запуск Python скрипта

#### Шаблон BAT-файла:

```bat
@echo off
chcp 65001 > nul
echo ========================================
echo Запуск [Название Бота]
echo ========================================
echo.

REM Активация виртуального окружения (если используется)
if exist venv\Scripts\activate.bat (
    call venv\Scripts\activate.bat
)

REM Запуск бота
python scripts\run_<bot_name>.py

REM Пауза при ошибке
if errorlevel 1 (
    echo.
    echo Ошибка при запуске бота!
    pause
)
```

#### Пример для Post FLOW:

```bat
@echo off
chcp 65001 > nul
if exist venv\Scripts\activate.bat call venv\Scripts\activate.bat
python -m blocks.post_flow.bot
if errorlevel 1 pause
```

Блок NewsBot (RSS) `blocks.spambot` из репозитория удалён.

## 📁 Правила структуры проекта

### Правило 2: Структура блоков

**Каждый блок должен иметь стандартную структуру**

```
blocks/
└── block_name/              # Latin folder name
    ├── __init__.py          # Экспорт основных классов
    ├── основной_файл.py     # Основной код блока
    ├── README.md            # Полная документация
    ├── QUICK_START.md       # Быстрый старт (опционально)
    └── config.example.yaml  # Пример конфигурации (если нужно)
```

#### Требования:

- ✅ **Названия папок блоков — только латиница.** Например: `post_flow`, `ai_integrations`, `autopost_zen`. Кириллица в путях не допускается (проблемы с CMD/скриптами).
- ✅ `__init__.py` должен экспортировать основные классы
- ✅ `README.md` должен содержать полную документацию
- ✅ Основной файл должен быть назван по имени блока
- ✅ Все зависимости должны быть добавлены в `requirements.txt`

## 💾 Правила работы с бекапами

### Правило 3: Обязательные бекапы

**Создавать бекап ОБЯЗАТЕЛЬНО в следующих случаях:**

1. ✅ **Блок работает** - первая рабочая версия
   ```bash
   python scripts/backup_manager.py create blocks/block_name/file.py "Первая рабочая версия" --tags working,v0.1
   ```

2. ✅ **Перед крупными изменениями** - рефакторинг, переписывание
   ```bash
   python scripts/backup_manager.py create blocks/block_name/file.py "Перед рефакторингом" --tags before-refactor
   ```

3. ✅ **Стабильная версия** - полностью протестирована
   ```bash
   python scripts/backup_manager.py create blocks/block_name/file.py "Стабильная версия" --tags stable,v1.0
   ```

4. ✅ **Перед коммитом в Git** - финальная версия
   ```bash
   python scripts/backup_manager.py create blocks/block_name/file.py "Версия для коммита" --tags production,v1.0
   ```

#### НЕ создавать бекап:

- ❌ После каждой мелкой правки
- ❌ Если блок еще не работает
- ❌ При минимальных изменениях

## 📝 Правила документации

### Правило 4: Обязательная документация

**Каждый блок должен иметь документацию**

#### Минимальные требования:

1. **README.md** в директории блока:
   - Описание блока
   - Установка и настройка
   - Примеры использования
   - API Reference
   - Troubleshooting

2. **Docstrings** в коде:
   - Для всех классов
   - Для всех публичных методов
   - Для сложных функций

3. **Примеры использования** в README:
   - Базовое использование
   - С конфигурацией
   - Интеграция в пайплайн

#### Шаблон docstring:

```python
def method_name(self, param1: str, param2: int) -> bool:
    """
    Краткое описание метода
    
    Args:
        param1: Описание параметра 1
        param2: Описание параметра 2
    
    Returns:
        Описание возвращаемого значения
    
    Raises:
        ValueError: Когда возникает эта ошибка
    
    Example:
        >>> obj.method_name("test", 42)
        True
    """
    pass
```

## 🔧 Правила конфигурации

### Правило 5: Переменные окружения и хранение ключей

**Все секреты и токены ТОЛЬКО через переменные окружения**

#### Требования:

1. ✅ Использовать `.env` файл для локальной разработки
2. ✅ Добавить пример в `.env.example`
3. ✅ НЕ коммитить `.env` в Git
4. ✅ Документировать все переменные в `KEYS_AND_TOKENS.md`
5. ✅ Группировать ключи по сервисам с комментариями

#### Централизованное хранение:

**Все ключи и токены документированы в одном месте:**
- 📄 [KEYS_AND_TOKENS.md](KEYS_AND_TOKENS.md) - полная документация всех ключей
- 📄 [.env.example](.env.example) - шаблон с примерами
- 📄 `.env` - реальные ключи (НЕ коммитится!)

#### Пример:

```python
import os
from dotenv import load_dotenv

load_dotenv()

API_KEY = os.getenv("API_KEY")
if not API_KEY:
    raise ValueError("API_KEY must be set in environment variables")
```

#### Запрещено:

```python
# ❌ НЕПРАВИЛЬНО - хардкод токенов
API_KEY = "1234567890:ABCdefGHI..."

# ❌ НЕПРАВИЛЬНО - токены в коде
BOT_TOKEN = "your_token_here"
```

#### Структура .env файла:

```env
# ============================================
# Название сервиса (назначение)
# Где получить: ссылка
# ============================================
SERVICE_API_KEY=your_key_here
SERVICE_API_URL=https://api.service.com
```

## 🎯 Правила именования

### Правило 7: Соглашения по именованию

#### Файлы и директории:

- **blocks:** названия папок латиницей (например, `ai_integrations`, `autopost_zen`)
- **Python файлы:** `snake_case` (например, `grs_ai_client.py`, `scheduler.py`)
- **Классы:** `PascalCase` (например, `GRSAIClient`)
- **Функции/методы:** `snake_case` (например, `send_post`, `fetch_feed`)
- **Константы:** `UPPER_SNAKE_CASE` (например, `MAX_RETRIES`, `API_URL`)

#### Переменные окружения:

- **Формат:** `UPPER_SNAKE_CASE`
- **Префикс:** По сервису (например, `TELEGRAM_`, `GRS_AI_`)
- **Примеры:** `TELEGRAM_BOT_TOKEN`, `GRS_AI_API_KEY`

## 📋 Правила коммитов

### Правило 8: Формат коммитов

**Использовать понятные сообщения коммитов**

#### Формат:

```
<тип>: <краткое описание>

<подробное описание (опционально)>
```

#### Типы коммитов:

- `feat:` - новая функциональность
- `fix:` - исправление бага
- `docs:` - изменения в документации
- `refactor:` - рефакторинг кода
- `test:` - добавление тестов
- `chore:` - рутинные задачи (обновление зависимостей и т.д.)

#### Примеры:

```
feat: добавлен блок NewsBot для автопостинга в Telegram

Реализован RSS бот с автоматическим переводом и фильтрацией.
Добавлена документация и скрипт запуска.

fix: исправлена обработка кодировки в GRS AI Client

refactor: улучшена структура конфигурации NewsBot

docs: обновлена документация по системе бекапов
```

## 🔒 Правила безопасности

### Правило 9: Безопасность

**Никогда не коммитить секреты в Git**

#### Запрещено коммитить:

- ❌ `.env` файлы
- ❌ Токены и API ключи
- ❌ Пароли
- ❌ Приватные ключи
- ❌ Credentials файлы

#### Разрешено коммитить:

- ✅ `.env.example` с примерами
- ✅ Документацию
- ✅ Конфигурационные файлы без секретов

#### Проверка перед коммитом:

```bash
# Проверить, что .env в .gitignore
cat .gitignore | grep .env

# Проверить, что нет секретов в staged файлах
git diff --cached | grep -i "token\|key\|password"
```

## 📊 Правила логирования

### Правило 10: Логирование

**Использовать стандартный модуль logging**

#### Требования:

1. ✅ Использовать `logging` вместо `print`
2. ✅ Настраивать уровни логирования
3. ✅ Использовать понятные сообщения

#### Уровни логирования:

- `DEBUG` - детальная отладочная информация
- `INFO` - общая информация о работе
- `WARNING` - предупреждения
- `ERROR` - ошибки
- `CRITICAL` - критические ошибки

#### Пример:

```python
import logging

logger = logging.getLogger(__name__)

# Настройка
logging.basicConfig(
    format="%(asctime)s [%(levelname)s] %(message)s",
    datefmt="%Y-%m-%d %H:%M:%S",
    level=logging.INFO
)

# Использование
logger.info("Бот запущен")
logger.warning("Не удалось загрузить ленту")
logger.error("Ошибка отправки поста: %s", error)
```

## 🎨 Правила кода

### Правило 13: Стиль кода

**Следовать PEP 8 и использовать type hints**

#### Основные правила:

1. ✅ Использовать type hints для параметров и возвращаемых значений
2. ✅ Максимальная длина строки: 100 символов
3. ✅ Использовать f-strings для форматирования
4. ✅ Избегать глобальных переменных

#### Пример:

```python
from typing import Optional, List, Dict

def process_data(
    items: List[str],
    config: Optional[Dict[str, any]] = None
) -> bool:
    """
    Обработка данных
    
    Args:
        items: Список элементов для обработки
        config: Опциональная конфигурация
    
    Returns:
        True если успешно
    """
    if config is None:
        config = {}
    
    for item in items:
        logger.info(f"Обработка: {item}")
    
    return True
```
