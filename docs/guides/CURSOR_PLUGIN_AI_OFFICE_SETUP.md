# Cursor Plugin "AI Office": полная инструкция по настройке и работе

## 1) Что вы получите

Этот гайд показывает, как собрать у себя автономный плагин в Cursor в стиле "директор + субагенты", который:

- принимает тему страницы/лонгрида;
- параллельно делает SEO и ресерч;
- генерирует структуру и текст;
- собирает HTML/CSS-блоки под стиль сайта;
- публикует страницу;
- прогоняет визуальный и SEO/GEO-контроль;
- возвращает ссылку + итоговый отчёт.

Идея: вы запускаете одну задачу, дальше система идёт по рельсам почти без ручного вмешательства.

---

## 2) Архитектура решения

Ниже минимальная рабочая модель.

1. **Plugin-слой (Cursor Plugin)**  
   Содержит роли, правила, команды, скиллы и orchestration-логику.

2. **Tool-слой (MCP)**  
   Даёт агентам инструменты: генерация текста/изображений, публикация, ресерч, браузерные действия.

3. **Execution-слой (проект)**  
   Скрипты/модули, которые реально собирают страницу, публикуют и валидируют результат.

4. **Общий контекст задачи (handoff/state)**  
   Единый артефакт, через который агенты передают результат друг другу (плюс отчёты QA).

---

## 3) Роли агентов (референсная схема)

Ниже схема, очень близкая к "офисной" модели из вашего сценария.

| Роль | Задача | Вход | Выход |
|---|---|---|---|
| Director | Оркестрация пайплайна и контроль циклов | Бриф/тема | План шагов, запуск/остановка стадий |
| Kolya SEO | Семантика, частотность, meta, H-структура | Тема + Wordstat/SEO tools | SEO-core, title/description, H1-H3 |
| Artem Research | Сбор фактов и свежих источников | Тема + web tools | Проверенные факты и источники |
| Zhenya Writer | Черновик лонгрида/лендинга | SEO + ресерч | Текстовые секции 8k–20k |
| Artur Ads | Нативные рекламные/CTA-вставки | Текст + рекламные правила | CTA-блоки и точки встройки |
| Alina Hero | Hero-блок (визуал+смысл) | Позиционирование страницы | Hero section |
| Visual Mid Agent | Доп. визуальные "крючки" в теле | Структура страницы | Mid-page engagement block |
| Natasha Layout | Вёрстка и сборка целой страницы | Все контент-блоки | Готовый HTML/CSS/JS пакет |
| Yura Publish | Публикация на хостинг/CMS | Build-артефакты | URL опубликованной страницы |
| Max Visual QA | Визуальная проверка (скриншоты/сдвиги) | URL | QA-репорт по верстке |
| Lyonya GEO/SEO QA | Проверка индексации и GEO-ready признаков | URL + HTML | SEO/GEO чеклист и правки |

## 5) Шаг 1. Подготовить MCP-инфраструктуру

### 5.1 Подключить сервер `content-factory`

Используйте шаблон:

```json
{
  "mcpServers": {
    "content-factory": {
      "command": "python",
      "args": ["-m", "blocks.mcp_server"],
      "cwd": "/path/to/ContentZavod"
    }
  }
}
```

Что вы получаете из коробки:

- `grs_chat`, `grs_chat_messages`, `grs_models`, `grs_image`
- `zen_publish`, `zen_draft`, `zen_list_articles`

Если нужен новый инструмент (например, Wordstat API), добавляйте его в:

- `blocks/mcp_server/server.py` с декоратором `@mcp.tool()`.

### 5.2 Добавить браузерный MCP для QA/ресерча

Для навигации и визуальной проверки подключают Playwright MCP (официальный).  
Подробности: `docs/guides/MCP_BROWSER.md`.

## 7) Шаг 3. Описать data contract между агентами

Главный принцип: каждый агент пишет **строго в свой раздел** общего state-файла (или в отдельный файл), а не перезаписывает всё подряд.

### Вариант A (рекомендуется): отдельные файлы по ролям

```text
run_state/
  run-<id>/
    00_director_plan.md
    10_seo_kolya.md
    11_research_artem.md
    20_writer_zhenya.md
    21_ads_artur.md
    30_hero_alina.md
    31_visual_mid.md
    40_layout_natasha.md
    50_publish_yura.md
    60_visual_qa_max.md
    61_geo_seo_qa_lyonya.md
    99_final_report.md
```

### Вариант B: один handoff-файл

Если используете единый `handoff.md`, введите правила:

- каждый агент обновляет только свой маркерный блок;
- запись только после "lock acquired";
- директор сверяет контрольную сумму секций после каждой стадии;
- конфликт записи = автоматический retry с backoff.

## 8) Шаг 4. Настроить оркестратор (Director)

Director должен работать как конечный автомат, а не "свободный чат".

### Состояния пайплайна

1. `INIT`
2. `SEO_RESEARCH_PARALLEL`
3. `WRITE_CONTENT`
4. `ADS_INSERTION`
5. `VISUAL_BLOCKS_PARALLEL`
6. `LAYOUT_BUILD`
7. `PUBLISH`
8. `VISUAL_QA`
9. `GEO_SEO_QA`
10. `FIX_LOOP` (если QA fail)
11. `DONE`

### Что должен делать Director на каждой стадии

- проверять входные артефакты;
- запускать нужных субагентов;
- фиксировать статус `pending/in_progress/done/fail`;
- при `fail` запускать ограниченный цикл исправлений (например, до 3 попыток);
- формировать финальный отчёт.

---

## 9) Шаг 5. Типовой E2E-цикл (как "офис" работает в бою)

```text
1) Director очищает состояние прошлого запуска
2) Kolya + Artem идут параллельно (SEO + ресерч)
3) Zhenya пишет основной текст по входным данным
4) Artur вставляет CTA/рекламные блоки
5) Alina + Visual Mid делают визуальные секции
6) Natasha собирает страницу (HTML/CSS/JS)
7) Yura публикует страницу и отдаёт URL
8) Max проверяет визуал, делает скриншоты, выдаёт баг-репорт
9) Lyonya делает SEO/GEO аудит (индексация, мета, schema, AI-crawl readiness)
10) Если есть ошибки -> FIX_LOOP -> повтор QA
11) Если всё ок -> финальный отчёт + ссылка
```

---

## 10) Шаг 6. Настроить качество и автоисправления

### 10.1 Visual QA gate (Max)

Проверки:

- "поехавшие" блоки;
- переполнение текста;
- контраст/читаемость;
- целостность hero и CTA;
- базовая desktop/mobile проверка.

Результат:

- `pass` или `fail`;
- список конкретных блоков для фикса;
- ссылки на скриншоты.

### 10.2 GEO/SEO QA gate (Lyonya)

Проверки:

- title/description/H1 соответствуют семантике;
- корректные canonical/robots/sitemap signals;
- schema.org (если используете);
- indexability (нет случайных noindex);
- сигналы для AI-поиска (структура, фактичность, источники).

Результат:

- чеклист с приоритетами `critical/major/minor`;
- набор точечных правок для возврата в FIX_LOOP.

## 11) Шаг 7. Контроль бюджета и токенов

Чтобы удерживать стоимость на низком уровне:

1. Держите "тяжёлую" модель только для сложных стадий (например, Writer + финальный Director audit).
2. Для рутинных шагов (форматирование, короткие правки) используйте более дешёвую модель.
3. Пишите короткие role-specific промпты без дублирования контекста.
4. Не передавайте весь пайплайн каждому агенту — только релевантные артефакты.
5. Лимитируйте FIX_LOOP (например, максимум 2-3 цикла).

## 13) Шаг 9. Набор метрик, которые стоит фиксировать

На каждый запуск сохраняйте:

- общее время цикла;
- количество FIX_LOOP итераций;
- итог `pass/fail`;
- расходы токенов по стадиям;
- количество критических QA-ошибок;
- базовые SEO-показатели (title/meta/H-структура);
- технические метрики страницы (например, Lighthouse/PageSpeed).

Это даст объективный контроль качества, а не "ощущение, что стало лучше".

## 14) Частые проблемы и решения

| Проблема | Причина | Решение |
|---|---|---|
| Субагенты перезаписывают друг друга | Общий файл без изоляции | Перейти на per-agent файлы или lock-механику |
| Цикл зависает на подтверждении | Неполные правила автономности | В rules зафиксировать auto-continue до QA gate |
| Плохое качество текста | Writer без качественного ресерча | Усилить вход от SEO + Research, добавить source validation |
| Страница "плывёт" после публикации | Разница окружений staging/prod | Ввести visual QA после публикации, не до неё |
| Дорого по токенам | Слишком тяжёлая модель на всех шагах | Разделить модели по ролям и лимитировать контекст |

## 15) Рекомендуемый roadmap внедрения

1. **MVP**: Director + 4 агента (SEO, Research, Writer, Publish).
2. Добавить **Layout + Visual QA**.
3. Добавить **GEO/SEO QA + FIX_LOOP**.
4. Добавить визуальные агенты и рекламные вставки.
5. Оптимизировать токены и время цикла по метрикам.

Так вы получите стабильность раньше, чем если сразу запускать 10+ ролей без контрактов.

---

## 16) Как понять, что система "встала на рельсы"

Считайте систему зрелой, когда:

- 3+ запуска подряд проходят без ручных вмешательств;
- QA-ошибки исправляются в пределах лимита FIX_LOOP;
- выход стабильно пригоден к публикации;
- время и стоимость запуска предсказуемы.
